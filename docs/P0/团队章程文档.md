# CampusHub 项目 AI 工具链选型原因

**团队名称：** 倒地爆冲

本团队在综合评估了项目需求与各 AI 工具的特性后，决定采用**“多工具互补组合”**的策略，以应对敏捷开发不同阶段的复杂场景。具体选型与评估如下：

## 1. AI 编程助手 (评估候选：Cursor, Antigravity)
* **最终决策：两者结合使用**
* **选择 Cursor 的理由**：
  * **场景分工**：用于从零搭建核心模块或进行大规模代码重构。
  * **评估**：Cursor 作为深度定制的 AI IDE，其全局代码库理解能力（Codebase Context）极为强大。在 P4 核心编码阶段，它能极大提升整文件生成和复杂逻辑编写的效率。
* **选择 Antigravity 的理由**：
  * **场景分工**：用于日常轻量级开发、快速查错与局部代码补全。
  * **评估**：作为轻量级插件，它能无缝集成到团队现有的 VS Code 环境中。易用性极高，团队成员无需改变原有习惯，适合在修改小 Bug 或查阅 API 文档时快速唤起。

## 2. AI 对话工具 (评估候选：Gemini, ChatGPT)
* **最终决策：两者结合使用**
* **选择 Gemini 的理由**：
  * **场景分工**：主攻 P1 需求分析阶段的长文档处理与架构设计辅助。
  * **评估**：Gemini 拥有超大的上下文窗口（Context Window）。我们可以在需求分析阶段，将所有的用户调研问卷、访谈记录以及 IEEE 830 标准要求一次性喂给它，它不会遗忘上下文，非常适合生成长篇文档。
* **选择 ChatGPT 的理由**：
  * **场景分工**：主攻 P3 详细设计与 P4 编码阶段的逻辑推导与 Debug。
  * **评估**：ChatGPT 在代码逻辑推理、算法设计和报错信息解析方面表现卓越。当遇到复杂的业务逻辑（如订单状态机流转）或死锁报错时，其强大的逻辑思辨能力能提供最准确的解决方案。

## 3. AI 代码审查工具 (评估候选：CodeRabbit, AI Code Review)
* **最终决策：两者结合使用**
* **选择 CodeRabbit 的理由**：
  * **场景分工**：作为 GitHub 仓库的主力 PR (Pull Request) 审查员。
  * **评估**：开箱即用，集成成本极低。团队成员每次发起合并请求时，CodeRabbit 会自动进行逐行代码审查，并给出优化建议，大幅减轻团队“开发负责人”的人工 Review 负担。
* **选择 AI Code Review (脚本/Action类) 的理由**：
  * **场景分工**：作为 CI 流水线中的底层安全与规范拦截器。
  * **评估**：相比于 CodeRabbit 的业务向审查，我们可以在 CI 中配置定制化的 AI Code Review 脚本，专门针对团队特定的代码规范（如变量命名规范、硬编码密码检查）进行硬性拦截。

## 4. AI 测试工具 (评估候选：CodiumAI, 通义灵码)
* **最终决策：选择 CodiumAI**
* **选择 CodiumAI 的理由**：
  * **场景分工**：用于 P6 质量保障阶段生成单元测试用例。
  * **评估**：它是专门针对测试场景优化的 AI 工具，不仅能自动生成覆盖正常流的测试，还能主动分析代码并生成边界条件（Edge Cases）的测试用例，功能专一且强大。
* **不选择通义灵码的理由**：
  * 虽然其功能全面，但我们已经有了 Cursor 和 Antigravity 作为编程助手，为了避免工具链过于臃肿，我们倾向于在测试环节使用垂直领域更专业的 CodiumAI。
